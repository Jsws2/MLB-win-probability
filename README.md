# MLB 실시간 승률 예측 시스템

MLB 경기를 보다가 "지금 이 상황에서 어느 팀이 얼마나 유리한 거지?"라는 궁금증에서 출발한 프로젝트입니다.
단순히 현재 점수만 보는 게 아니라, 이닝/아웃카운트/주자 상황까지 종합해서 홈팀 승리 확률을 수치로 보여주는 게 목표였습니다.

## 1. 프로젝트 목표

야구에서 "어느 팀이 이길 것 같다"는 판단은 보통 현재 점수 차이에 의존하는 경우가 많습니다. 그런데 점수만으로는 경기 상황을 제대로 설명하기 어렵습니다. 예를 들어, 1회 초에 2점 뒤지고 있는 것과 9회 말에 2점 뒤지고 있는 것은 실제 승률 측면에서 전혀 다른 상황입니다. 1루 만루에서 1점 차인 경우와 주자 없이 1점 차인 경우도 마찬가지입니다.

이 프로젝트는 이런 문제의식에서 출발했습니다. 현재 경기 상황 전체를 수치화한 뒤 로지스틱 회귀 모델에 입력해, 홈팀이 이길 확률을 실시간으로 계산하고 시각화합니다. 단순 점수 확인이 아닌, 이닝/공격 순서/아웃카운트/점수차/주자 상황을 종합해 확률로 표현한다는 점이 핵심입니다.

## 2. 시스템 구조

```
MLB Stats API
     ↓
경기 데이터 수집  (collector.py)
     ↓
Feature 생성     (features.py)
  inning, is_top, outs, score_diff, runner_1b/2b/3b
     ↓
Logistic Regression  (model.py)
     ↓
승률 계산  P(홈팀 승리) ∈ [0, 1]
     ↓
Tkinter Dashboard 시각화  (main.py)
```

| 단계 | 파일 | 역할 |
|------|------|------|
| 데이터 수집 | `collector.py` | MLB Stats API 호출, 경기 일정 및 타석별 이벤트 수집 |
| Feature 생성 | `features.py` | JSON 응답에서 7개 Feature 벡터 추출 |
| 모델 추론 | `model.py` | 학습된 Pipeline으로 승률 계산 |
| 시각화 | `main.py` | 승률 변화 그래프, 타석별 상황 카드 렌더링 |

## 3. 데이터 수집

MLB Stats API(`statsapi.mlb.com`)는 별도 API 키 없이 사용할 수 있는 공개 API입니다. `requests` 라이브러리로 HTTP GET 요청을 보내 JSON 응답을 파싱하는 방식으로 구현했습니다.

**경기 일정 조회**

```
GET https://statsapi.mlb.com/api/v1/schedule?sportId=1&date=2026-06-10
```

날짜별 경기 목록과 각 경기의 고유 ID(`gamePk`)를 반환합니다.

**타석별 실시간 데이터 조회**

```
GET https://statsapi.mlb.com/api/v1/game/{gamePk}/feed/live
```

해당 경기의 전체 타석 이벤트가 담긴 JSON을 반환합니다. 각 타석에는 이닝, 공격/수비 여부, 아웃카운트, 주자 위치, 타격 결과 등이 포함되어 있습니다.

이 응답에서 타석 단위로 상태를 추출해 Feature 벡터를 생성합니다. 경기 종료 후에는 최종 승패 라벨(`home_wins`)도 함께 기록해 학습 데이터로 활용합니다.

학습 데이터는 2023년 4월부터 9월까지 정규시즌 약 200경기를 수집했습니다. 연장전 동점으로 종료된 경기는 승패 라벨을 만들 수 없어 학습에서 제외했습니다.

## 4. Feature Engineering

### Feature 선택 기준

처음에는 볼카운트, 타자 이름, 구종 등 더 많은 정보를 Feature로 쓰는 방안도 검토했습니다. 그러나 Feature 수가 늘어나면 학습 데이터 대비 변수가 많아져 과적합(Overfitting) 위험이 커지고, 모델이 어떤 근거로 확률을 계산하는지 해석하기 어려워집니다.

이 프로젝트는 경기 흐름을 설명하는 데 실질적으로 중요한 변수만 남기는 방향으로 Feature를 설계했습니다. 타자/투수 개인 능력치는 배제하고, 누구나 납득할 수 있는 경기 상황 지표 7개만 사용합니다.

### Feature 목록

| Feature | 설명 | 처리 방식 |
|---------|------|-----------|
| `inning` | 현재 이닝 | 연장전은 9로 cap (학습 범위 통일) |
| `is_top` | 초공격(1) / 말공격(0) | 같은 점수차라도 공격/수비에 따라 유불리 다름 |
| `outs` | 아웃카운트 | 0, 1, 2 |
| `score_diff` | 홈팀 점수 - 원정팀 점수 | -10 이상 +10 이하로 clip |
| `runner_1b` | 1루 주자 여부 | 0 또는 1 |
| `runner_2b` | 2루 주자 여부 | 0 또는 1 |
| `runner_3b` | 3루 주자 여부 | 0 또는 1 |

**입력 예시** — 9회말 1아웃, 홈팀이 1점 뒤지고 있고 2루에 주자가 있는 상황:

```python
[9, 0, 1, -1, 0, 1, 0]
# inning=9, is_top=0(말), outs=1, score_diff=-1, runner_2b=1
```

이 벡터 하나가 모델의 입력이 되고, 모델이 홈팀 승리 확률 P를 반환합니다.

## 5. 모델 선택

### Logistic Regression을 선택한 이유

"홈팀이 이기는가, 지는가"는 이진 분류(Binary Classification) 문제입니다. 이 구조에 맞는 여러 알고리즘 중 Logistic Regression을 최종 선택했습니다. 이유는 두 가지입니다.

첫 번째는 확률 출력입니다. `predict_proba`를 통해 0과 1 사이의 확률값을 직접 뽑을 수 있습니다. 단순히 "이긴다/진다"가 아니라 "현재 상황에서 이길 확률이 63%다"라는 수치를 화면에 보여줘야 하기 때문에, 확률을 자연스럽게 출력하는 모델이 필요했습니다.

두 번째는 설명 가능성(Explainability)입니다. Logistic Regression은 각 Feature의 계수(coefficient)가 그대로 영향도로 해석됩니다. 어떤 Feature가 승률을 올리고 내리는지 계수만 보면 알 수 있어서, 모델이 "왜 이 확률을 냈는지" 설명하기가 수월합니다.

Random Forest와 SVM도 검토했지만, 두 알고리즘은 내부 결정 과정이 불투명하거나 확률 보정이 별도로 필요해서 이 프로젝트의 목적에는 맞지 않았습니다.

### 학습 파이프라인

```python
pipeline = Pipeline([
    ("scaler", StandardScaler()),
    ("clf", LogisticRegression(C=0.5, max_iter=2000, solver="lbfgs", random_state=42)),
])
```

---

## 6. 데이터 전처리

`StandardScaler`를 Pipeline 첫 단계에 넣어 모든 Feature를 평균 0, 표준편차 1로 정규화합니다.

이 과정이 필요한 이유는 Feature 간 스케일 차이 때문입니다. `inning`은 1에서 9 사이, `score_diff`는 -10에서 +10 사이처럼 변수마다 범위가 다릅니다. 정규화 없이 학습하면 스케일이 큰 변수의 계수가 과도하게 커지거나, lbfgs 솔버의 수렴이 불안정해질 수 있습니다. `StandardScaler`를 통해 이 문제를 방지하고 최적화 과정을 안정화합니다.

`C=0.5`는 L2 정규화 강도의 역수로, 기본값(1.0)보다 정규화를 강하게 줘서 특정 Feature에 과도하게 의존하는 과적합을 억제합니다.

## 7. 학습 및 검증

### 데이터 분할

```python
X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.2, stratify=y, random_state=42
)
```

수집한 데이터를 80:20 비율로 나눠 학습셋과 검증셋을 구성했습니다. `stratify=y`를 적용해 홈팀 승/패 비율이 학습셋과 검증셋에서 동일하게 유지되도록 했습니다. 이를 적용하지 않으면 한쪽 클래스가 편중될 수 있고, 특히 데이터 규모가 크지 않은 경우 검증 결과가 불안정해질 수 있습니다.

데이터 분할 시에는 데이터 누수(Data Leakage)가 발생하지 않도록, `StandardScaler`의 `fit`은 학습셋에만 수행하고 검증셋에는 `transform`만 적용합니다. Pipeline 구조가 이 과정을 자동으로 보장합니다.

### 최종 검증

학습/검증 분할과 별개로, 학습에 전혀 사용하지 않은 최근 20경기(1,529타석)로 최종 성능을 검증했습니다. 학습 데이터와 시기가 겹치지 않는 경기를 사용해 실제 예측 환경과 가까운 조건에서 평가했습니다.

## 8. 성능 평가

### 평가 지표 선택 이유

Accuracy 하나만으로는 확률 예측 모델의 품질을 제대로 평가하기 어렵습니다. 예를 들어, 모든 경기를 "홈팀 승리"로 예측해도 홈팀 승률이 53% 수준이면 Accuracy가 0.53 이상 나옵니다. 이런 문제 때문에 세 가지 지표를 함께 사용했습니다.

| 지표 | 의미 | 결과 |
|------|------|------|
| **Accuracy** | 전체 타석 중 승리 예측이 맞은 비율 | 0.796 |
| **ROC-AUC** | 임계값과 무관하게 확률 순위가 얼마나 잘 맞는지 | 0.884 |
| **Brier Score** | 예측 확률이 실제 결과에 얼마나 가까운지 (낮을수록 좋음) | 0.136 |

### 결과 해석

Accuracy 0.796은 약 80%의 타석에서 최종 승리팀을 맞혔다는 의미입니다. ROC-AUC 0.884는 무작위 예측(0.5) 대비 확률 순위 구분 능력이 높다는 것을 보여줍니다. Brier Score 0.136은 예측 확률 자체가 실제 결과와 잘 보정되어 있다는 것을 나타냅니다. 단순히 "이긴다/진다"를 맞히는 것이 아니라, "이길 확률이 70%"라고 말했을 때 실제로 그 상황의 70% 정도가 승리로 이어진다는 의미입니다.

## 9. Feature 계수 분석

Logistic Regression의 계수(coefficient)를 분석해 각 Feature가 홈팀 승률에 미치는 영향을 확인했습니다.

분석 결과 `score_diff`의 계수 절댓값이 가장 컸습니다. 점수 차이가 승패 예측에 가장 큰 영향을 미친다는 직관과 일치합니다. `is_top`은 두 번째로 영향이 컸는데, 같은 점수차라도 초공격(원정팀 공격)과 말공격(홈팀 공격)이 승률에 미치는 방향이 다르다는 점이 반영된 결과입니다. 주자 상황(`runner_1b/2b/3b`) 역시 유의미하게 기여했으며, 이닝(`inning`)은 경기 후반일수록 점수차의 영향이 커진다는 맥락과 연관되어 있습니다.

## 10. 구현 과정에서 어려웠던 점

**실시간 데이터 처리**
MLB API의 `feed/live` 응답은 경기 중에도 계속 갱신됩니다. LIVE 경기에서 새 타석이 추가됐는지 주기적으로 확인(polling)하고, 추가된 경우에만 Feature를 다시 생성해 예측을 갱신하는 구조를 만드는 데 신경을 많이 썼습니다.

**점수 집계 처리**
초기 구현에서는 `result.rbi` 값으로 점수를 집계했는데, 폭투나 포일처럼 타점 없이 득점하는 케이스에서 실제 점수와 집계 값이 어긋나는 문제가 있었습니다. API 응답의 `runners[].movement.end == "score"` 필드를 기준으로 실제 득점한 주자 수를 세는 방식으로 수정해 해결했습니다.

**주자 상태 추적**
타석이 시작될 때의 주자 상태와 타석이 끝난 후의 주자 상태를 구분해야 했습니다. `runners[].movement.originBase` 필드를 활용해 타석 시작 시점의 주자 위치를 기준으로 Feature를 생성했습니다.

**이닝 처리**
연장전이 있는 경기에서 이닝이 10회 이상으로 올라가는데, 학습 데이터 대부분이 정규 9이닝 데이터라 10회 이상을 그대로 Feature로 쓰면 학습 범위를 벗어납니다. 연장전 이닝은 9로 cap 처리해 학습 데이터와 범위를 맞췄습니다.


## 12. 주요 기능

- **날짜 선택 및 경기 목록 표시**: 날짜를 선택하면 해당 날 MLB 경기 목록이 출력됩니다.
- **타석별 승률 변화 그래프**: 경기를 선택하면 타석이 진행될수록 홈팀 승률이 어떻게 바뀌었는지 꺾은선 그래프로 확인할 수 있습니다.
- **최고 하락 / 최고 상승 장면 이동**: 경기 전체에서 승률이 가장 크게 변한 타석으로 바로 이동하는 버튼입니다. 어떤 플레이가 경기 흐름을 가장 많이 바꿨는지 한눈에 볼 수 있습니다.
- **타석별 Feature 확인**: 현재 선택된 타석에 어떤 Feature 값이 입력됐는지 확인할 수 있어, 모델이 어떤 정보를 바탕으로 승률을 계산했는지 추적할 수 있습니다.
- **이닝별 R/H/E 라인스코어**: 이닝별 득점, 안타, 실책을 테이블로 표시합니다.
- **경기 전 예측**: 팀 시즌 승률, ERA, OPS, 선발투수 성적 등을 조합해 경기 시작 전 승률을 예측합니다.

## 13. 한계

가장 큰 한계는 선수 개인 능력치가 반영되지 않는다는 점입니다. 같은 9회말 동점 상황이라도 마운드에 올라온 투수가 누구인지, 타석에 선 타자가 누구인지에 따라 실제 확률은 크게 달라질 수 있습니다. 현재 모델은 이를 전혀 고려하지 않습니다.

구체적으로 반영하지 못한 요소들은 다음과 같습니다:

- 타자 OPS (출루율 + 장타율)
- 투수 ERA (평균 자책점)
- 선발투수 당일 컨디션 및 구종 구성
- 불펜 투수 상태 (피로도, 최근 등판 여부)
- 팀의 최근 경기 흐름 (연승/연패 여부)

## 14. 향후 개선 방향

**Feature 확장**
- 타자 OPS, 투수 ERA를 Feature에 추가해 선수 개인 능력치 반영
- 선발투수 시즌 성적 및 최근 5경기 ERA 반영
- 팀의 최근 10경기 승률을 Feature로 추가

**모델 비교**
- 현재 Logistic Regression과 Random Forest, XGBoost를 비교 실험해 성능 차이 확인
- 앙상블 기법 적용 가능성 검토

**데이터 규모 확대**
- 현재 2023년 약 200경기 데이터를 복수 시즌으로 확장
- 포스트시즌 데이터 별도 처리 방안 검토
